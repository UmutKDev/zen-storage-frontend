## Phase 1 implementation notes (2026-06-06)

- **D-P1.0 — Auth.js v5: UI-driven multi-step, `authorize` finalizes.** Credentials `authorize` is single-shot and can't model intermediate 2FA state, and `AuthenticationResponseModel` is only `{ SessionId, ExpiresAt }`. So the screens (`features/auth`) walk the backend flow via the generated factories — `loginCheck` (→ `HasTwoFactor`/`HasPasskey`) → `login` → (if 2FA) `verify2FA` with the login `X-Session-Id` (the §6 handoff) — to obtain a final `SessionId`, then call `signIn("credentials", { sessionId })`. `authorize` re-validates the id via `Account/Profile` (explicit header) before minting the JWT. Register/reset reuse the finalize step.
- **D-P1.1 — `boundaries/element-types` is OFF for `features/**`.** eslint-plugin-boundaries matches element patterns with micromatch `{ contains: true }`, so the global utility patterns (`components/**`, `lib/**`, `hooks/**`, …) also match a feature's INTERNAL folders (`features/auth/components`, …) and misclassify them as the global element. There is no pattern-only anchor under `contains`. So element-types is disabled for files under `features/**` (intra-feature layering was permissive anyway). The critical guarantees all remain: cross-feature **hard barrel** (`entry-point` on `feature` + the `@/features/*/!(index)` import ban), no raw HTTP, no `@/service/generates`, and the leaf/one-way rules whose FROM files are top-level (still checked). The narrow `subfeature` element (for nested `features/storage/{browse,upload}`) is added in P3.
- **D-P1.2 — Split Auth.js config (edge/proxy-safe).** `lib/auth/config.ts` exports the base `authConfig` (no providers, no node deps); the proxy builds `NextAuth(authConfig).auth` from it to read the session JWT without pulling the factory→envelope→sonner chain. `lib/auth/nextauth.ts` (server-only) holds the full instance with the Credentials provider + `accountApiFactory`. Route protection lives in `lib/auth/proxy.ts` via the `auth((req) => …)` wrapper composed with the security headers; unauthenticated `(app)` → `/login?from=…`, authed `(auth)` → `/storage`.

## Phase 3 implementation notes (2026-06-07)

- **D-P3.1 — Phase 3 is staged in ~4 parts.** Storage core is the biggest phase, so it ships incrementally with a green + reviewed checkpoint between stages: **A** browse foundation (list/grid, breadcrumb deep-link, virtualization, usage bar, states) — DONE; **B** operations (create/rename/move/delete + multi-select/bulk + conflict dialog); **C** upload pipeline; **D** search/filter/sort + command palette + touch bottom-sheet. Each stage is planned + implemented on its own.
- **D-P3.2 — Upload transport = `UploadPart` proxy** (user choice, for Stage C). Chunks go through `cloudUploadApiFactory.uploadPart` (content-MD5; backend returns each part's ETag) rather than presigned direct-to-S3 PUT. Consequence: the "presigned S3 PUT" non-factory exception (CLAUDE rule #1 / data-layer §5) is **not exercised** — uploads stay 100% factory calls; no S3 CORS dependency.
- **D-P3.3 — `ListParts` absent → degraded resume (Stage C).** Verified there is **no `ListParts`** endpoint in the generated client or the NestJS upload controller (only Create/GetUrl(s)/UploadPart/Complete/Abort). The architecture's `ListParts`-based refresh-resume (upload-pipeline §6.2) + lost-ETag recovery (§6.4) can't be built verbatim. Stage C plan: persist the queue + per-part ETags to IndexedDB and resume via **idempotent part re-PUT** (S3 is idempotent on `(UploadId, PartNumber)`); if the local ETag record is lost, re-upload from part 1 (correct, just not optimal). Logged as a **backend gap** (add `ListParts`).
- **D-P3.4 — Browse breadcrumb derived from the URL, not fetched.** `BreadcrumbBar` builds crumbs from the `[[...path]]` segments (`toSegments`) — instant, no extra request, correct for normal folders. `Cloud/List/Breadcrumb` stays available if canonical display names (e.g. for renamed/encrypted folders) are needed later. Browse uses the **paginated** `listDirectories` + `listObjects` (not the combined unpaginated `list()`) so large folders virtualize off `Skip/Take/Count`; `itemsOf` normalizes the post-envelope `{items,count}` shape.
- **D-P3.5 — Query retry now skips 4xx.** `app/providers.tsx` retry was `retry: 2` (retried everything, contradicting its own comment + data-layer §2.9). Changed to a predicate: retry only when the error is **not** a typed `ApiError`, or its code is `SERVER_ERROR`/`NETWORK` (transient), up to 2× — never 4xx (auth/validation/conflict/404/429). Storage queries thread the TanStack `AbortSignal` into the factory `options` so fast folder-nav/unmount cancels in-flight GETs.
- **D-P3.7 — Browse uses single-call listing (no Skip/Take) + `delimiter: true` (found by LIVE verification, 2026-06-07).** Two browse bugs the mocked Vitest couldn't catch, both confirmed + fixed against the live backend:
  - **`delimiter: true` is required.** `CloudApi` `delimiter?: boolean` → backend `Delimiter`: true ⇒ list THIS folder level (subdirs as S3 common-prefixes + objects at the level); falsy ⇒ **recursive** listing. Stage A shipped without it. For objects the flag maps to the S3 `Delimiter` (`cloud.list.service` L189); for directories the level path always uses `/`.
  - **Skip/Take removed entirely** (user direction). The backend's **paginated** path (`usePagination = skip||take`) ignores `Take` for directories, returns overlapping S3-batch pages (a created folder rendered 2–3×), and its aggregation/continuation loop **errored on folder entry** ("Something went wrong"). Omitting **both** Skip and Take takes the backend's clean **non-paginated path** (`cloud.list.service` L341 dirs / L184 objects): one S3 call, the whole folder up to ~1000, no overlap, no error. So `getDirectories`/`getObjects` are single `useQuery`s (not `useInfiniteQuery`) calling `listDirectories/listObjects({ path, delimiter: true })`. Verified live: requests are `…?Path=<folder>&Delimiter=true` with **no `Skip`/`Take`**; subfolder entry loads with no error. `useFolderEntries` keeps a cheap key-dedupe as a guard; `lib/api/pagination.*` is now unused by browse (left as a general util) and `nextPage` stops on an empty page.
  - **Process note:** API-mocking unit tests can't catch request-param/contract bugs — **run the app against the live backend** for storage work. Backend gap (if pagination is ever needed for >1000-entry folders): honor `Take` + non-overlapping pages.
- **D-P3.8 — The Instance must NOT toast canceled requests (found by LIVE verification, 2026-06-07).** Threading the TanStack `AbortSignal` into the factory `options` (so fast folder-nav cancels stale GETs) exposed a latent bug: when TanStack aborts an in-flight request (navigation / Strict-Mode remount / refetch), axios throws `CanceledError` (`code: ERR_CANCELED`), and `service/interceptors/envelope.ts` mapped it to a `NETWORK` `ApiError` and **toasted "Something went wrong"** — 2–3 generic toasts on every folder entry (the canceled dirs + objects + usage requests). Fix: the rejected interceptor now **rethrows cancellations silently** (`error.code === "ERR_CANCELED" || error.name === "CanceledError"`) before any toast/sign-out; TanStack ignores aborted results. This was a global bug (any aborted request), not storage-specific — Phase 1/2 just never threaded a signal. Covered by a unit test (`tests/smoke/interceptor-cancel.test.ts`). Verified live: 0 toasts on folder entry/re-entry. `features/storage/operations` (sibling subfolder of `browse/`). **Stage B split B1→B2** (B1 = single-item via per-row `EntryActionsMenu` + header `CreateMenu`; B2 = multi-select + bulk + DnD). Payload conventions (from the old frontend + backend): create-folder `Path` has a trailing slash; create-file uses `{ Path: parent, Name }`; delete/move items are `{ Key: file.Path.Key | dir.Prefix, IsDirectory }`; move `DestinationKey` is `"/"` at root else `"<path>/"`; dir-rename `Path` drops the trailing slash. **`Cloud/Delete` handles files + unencrypted dirs in one call** (encrypted dirs need `Cloud/Directory/Delete` → Phase 5). **Conflict:** the Instance passes 409 through as `ApiError(code:CONFLICT)` carrying the body in `error.raw`; `extractConflictDetails` reads `error.raw.Status.Messages[0]` (`ConflictDetailsResponseModel`); the shared `useConflictMutation` retries with the chosen `ConflictStrategy` (REPLACE/KEEP_BOTH; SKIP cancels) — non-conflict errors are toasted centrally (D-P2.2). **Cache:** delete is optimistic (remove + reconcile via `invalidateFolder`; the settle-time refetch is the rollback); create/rename invalidate the folder; move invalidates the owner scope (source+dest). Move + Delete carry `Idempotency-Key` (fresh per attempt — a strategy-change retry is a new logical op; these mutations don't auto-retry). DTOs use generated models (`CloudDeleteModel`/`CloudMoveItemModel`, `ConflictResolutionModelStrategyEnum`). Reuse types deferred to Phase 5: 403-secure-folder→unlock routing in the op hooks + a hidden/folder-token slot in the `directories` query key.
- **D-P3.9 — Stage B2 batch semantics: one call per bulk action; partial-batch SKIP goes to the server (2026-06-10).** Bulk move/delete are **single** `Cloud/Move`/`Cloud/Delete` calls with `Items[]` (not per-item loops), so the 409 `ConflictDetailsResponseModel` (`ConflictCount`/`TotalItems`) covers the whole batch and **one strategy retry = apply-to-all** with the decided batch radius (one user action; a new batch starts fresh — no checkbox needed). Verified against `cloud.object.service.Move()`: conflicts are collected **before** any mutation (atomic 409) and a retried global strategy applies **only to the conflicted items** while the rest move normally — therefore `useConflictMutation.resolve("SKIP")` now **re-issues the op with `SKIP`** when `ConflictCount < TotalItems` (the non-conflicting rest must still move) and only cancels locally when **all** items conflict (preserves the single-item behavior). Selection: feature-local in-memory zustand (`operations/stores/selection.store.ts`; `selectedKeys` is the ⌘K contract); locked/concealed dirs are **excluded** from select/select-all/range/drag/drop (Phase 5 tokens); plain click selects **files** only (folders keep click-to-navigate); **modifier-click on folder links is hijacked for selection** (file-manager convention — open-in-new-tab is intentionally sacrificed). DnD: `MouseSensor` only (8px activation; touch = Stage D bottom sheet; Stage C file-drop uses native HTML5 drag events, no clash), dnd-kit `listeners` spread **without `attributes`** (they'd add `role="button"` inside a `listitem`), **no KeyboardSensor** (MoveDialog is the accessible move path — recorded with the Stage A a11y deferrals), a `suppressClickRef` swallows the synthetic post-drag click (else dropping over the origin folder link navigates), breadcrumb ancestors are drop targets, self/descendant drops are guarded client-side (`lib/dnd.ts`; backend 400 is the backstop). Bulk download = sequential presigns + hidden-anchor clicks with a 350 ms stagger, **files only** (multiple `window.open` after an await is popup-blocked; Chrome's one-time multi-download prompt is expected). **Latent bug fixed:** `useDelete`'s optimistic update still wrote the pre-D-P3.7 `InfiniteData` cache shape — it would throw on any populated plain-array cache (live), silently passing in mocked tests; rewritten to plain-array filters keyed off `entry.key` sets.
- **D-P3.10 — Stage C upload pipeline: proxy-transport realities + queue semantics (2026-06-11).** Implements §3.3 under D-P3.2/D-P3.3 with the following locked specifics, several of which **amend stale doc rows** (amendments recorded in the phase file's hardening block):
  - **Transport is 100% factory** (`cloudUploadApiFactory`): `Create → N × uploadPart (multipart/form-data, base64 `content-md5` via spark-md5 — Web Crypto has no MD5; MIT, +@types) → Complete / Abort`. The presign endpoints and the `presigned-put.ts` ESLint carve-out are **intentionally unused**.
  - **Part size 8 MiB** (`lib/upload/config.ts`): ≥ S3's 5 MiB non-last-part minimum, ≤ the proxy's 10 MB default cap; max file ≈ 80 GiB at the 10 000-part ceiling.
  - **Resolved-Key invariant:** Create's response `Key` (KEEP_BOTH renames server-side) is authoritative for every subsequent call + the persisted entry.
  - **Conflicts:** Create 409 → shared `ConflictPrompt` hosted by the tray; **SKIP is client-local cancel** (backend 400s SKIP for uploads); **apply-to-all radius = one user action** implemented as a per-batch remembered strategy (uploads are N independent Creates — unlike B2's single-call batches, the looped-batch checkbox from conflict-resolution §3 IS needed; `ConflictPrompt` grew an optional `applyToAll` slot).
  - **Degraded resume (no `ListParts`):** the immediately-persisted IndexedDB `partETags` record IS the resume state; missing parts re-PUT idempotently on `(UploadId, PartNumber)`; a lost record re-uploads from part 1. The doc's "ListParts 404 → session expired" probe is unimplementable — the backend surfaces S3 `NoSuchUpload` as a **generic 500** (no error mapping in `CloudS3Service.Send`), so session-gone degrades to ordinary retryable errors. Backend gaps logged: `ListParts` + `NoSuchUpload`→404 mapping. The persisted entry carries `ownerId` (restore filters by owner; team-ready), `partSize` (resume re-slices on original boundaries), and the Complete `idempotencyKey` (minted once, **reused across retries**; the interceptor honors pre-set keys).
  - **Abort is fire-once with a 3-attempt budget** then dropped (re-aborting → `NoSuchUpload` 500, so it is NOT safely repeatable as the doc claimed); `abortPending` entries retry on next load; the backend's orphan sweep owns escapes.
  - **Zero-byte files:** `Create(TotalSize: 0)` + ONE empty part + Complete (no single-PUT endpoint exists; stays factory).
  - **`MIME_MISMATCH` 409 does not exist in the backend** — no handler shipped; upload-pipeline §6.6 flagged stale.
  - **Quota:** client pre-flight from `StorageUsage` (incl. bytes already queued) blocks before start with upgrade hint; the backend's quota 400s carry only English message strings (no typed code) → recognized by `/storage limit/i` to halt the batch's remaining queued files. Complete can also 400 **after** upload (backend deletes the object) → terminal error.
  - **`suppressErrorToast`** axios-config flag (typed via `service/axios.d.ts` module augmentation, honored by the envelope interceptor): queue-managed calls skip the central toast — per-part retries would toast-storm; the tray owns error presentation. 401/403/409 behavior unchanged.
  - **Two drops stay structurally unambiguous:** OS-file drop = native HTML5 drag events (`FileDropZone`, depth-counter highlight, `types.includes("Files")` guard); item-move = dnd-kit MouseSensor — different event systems, no interplay. Folder upload = `webkitGetAsEntry` traversal + `webkitdirectory` picker, **desktop-only** (touch path = Stage D bottom sheet); dirs created shallowest-first with **create-409 = merge into existing**.
  - **Pause = drain** (in-flight parts finish; no scheduling of new parts), cancel = AbortController + server Abort, retry reuses surviving `partETags`. Engine is a module singleton (`upload/core/engine.ts`) writing to the in-memory `uploads.store`; sign-out teardown (`teardownUploads`) aborts everything and clears the tray, while owner-scoped IndexedDB entries survive for the owner's next session. IndexedDB is raw (≈60-line wrapper) — no new dep where the API suffices; `fake-indexeddb` (dev) covers tests.
