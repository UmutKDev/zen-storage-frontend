## Phase 0 implementation notes & deviations (2026-06-06)

These adjust the original P0 plan where the **as-built stack** required it. Each was needed for a green `tsc` + `lint` + `build`.

- **D-P0.1 ‚Äî `eslint-plugin-boundaries` pinned to `^5`.** The drop-in config in [folder-structure.md](../02-architecture/folder-structure.md) is v5-shaped; the installed v6 renamed `element-types`‚Üí`dependencies` and changed `entry-point` semantics. Pinned v5.4.0 to match the documented config. (Migrate to v6 `dependencies` rule in a later hardening pass.)
- **D-P0.2 ‚Äî `boundaries/entry-point` uses `default: "allow"`** (not `disallow`). Under v5, `default: disallow` with rules only for feature/subfeature blocks **every** non-feature import (lib, components, service). The intent (¬ß7 rule 2) is that ONLY features/subfeatures are barrel-locked; all other element types allow any entry file by path. Deep feature imports stay blocked via the explicit feature/subfeature rule **and** `no-restricted-imports`.
- **D-P0.3 ‚Äî `service/` may import `lib/`, and `axios` is allow-listed in `service/**`.** [data-layer.md ¬ß2.7](../02-architecture/data-layer.md) has the `Instance` composer + interceptors consume `lib/api` (`newIdempotencyKey`, `unwrapEnvelope`, `toastApiError`, `ApiError`) and `lib/i18n` (`t`). `service/` remains a leaf w.r.t. **features** (still disallowed). `service/Instance.ts` is the ONE sanctioned `axios` user; the feature-side `axios`/`fetch` ban is unchanged everywhere else.
- **D-P0.4 ‚Äî `Instance.baseURL` is the origin, NOT `origin + /Api`.** The generated client paths already embed `/Api` (e.g. `"/Api/Account/Profile"`). Appending `/Api` produced `/Api/Api/...`. `Instance` now uses `NEXT_PUBLIC_API_URL` with a defensive trailing-`/Api` strip. Supersedes data-layer ¬ß2.1's "`+ '/Api'`".
- **D-P0.5 ‚Äî Root `proxy.ts` defines `config.matcher` as a static literal** (re-exporting `config` from `lib/auth/proxy` fails Next 16.2's static analysis). Only the `proxy` **function** is re-exported; the matcher lives inline in the root shim.

Dark mode is **class-based** (`@custom-variant dark (&:is(.dark *))` + next-themes). Glass (`glass-overlay`, e3 elevation + highlight rim, reduced-transparency fallback) is applied to Dialog/Dropdown/Command overlays; primitives pulled via the shadcn MCP then wrapped.

The Phase-1 deferrals (passkey 1.2, legal/consent 1.6) are also done (D-P1.5). No deferred items remain.
- **D-P0.8 ‚Äî Security headers + privacy foundation (pass 2, 2026-06-06).**
  - **Headers in the proxy, not `next.config`** (security-headers ¬ß1): `lib/security/headers.ts` (`STATIC_HEADERS` + `buildCsp(nonce)` + `generateNonce`) and `lib/security/nonce.ts` (`getNonce`, RSC) compose into `lib/auth/proxy.ts`, so the per-request CSP nonce can be threaded.

Files over a 50 MiB cap, or a failed load, fall back to **open-in-new-tab** (direct nav to `Path.Url` works) + **download**. Two costs, both accepted: **(1) CSP** ‚Äî `frame-src` gains `https://view.officeapps.live.com` (a second deliberate loosening after D-P4.1's CDN; iframe is `sandbox="allow-scripts allow-same-origin allow-popups allow-forms allow-downloads"` + `referrerPolicy="no-referrer"`, cross-origin to the app so no escape). **`allow-forms` is required** ‚Äî the viewer POSTs a form to load its WOPI render frame; without it Chrome blocks the submission and the doc never renders. `allow-downloads` enables the viewer's download action. (Also depends on D-P4.6 ‚Äî COEP had to be removed for the cross-origin frame to load at all.) **(2) Privacy/KVKK (rule #13)** ‚Äî Microsoft fetches + renders the document server-side, so file **content egresses to a third party**. Mitigated: the `src` is a fresh **presigned** URL (not an app-auth URL), the viewer footer **discloses** "opened in Microsoft's online viewer" + always offers direct download, and the egress is documented in `privacy-compliance.md`. Office stays **best-effort**: any error/too-large/unsupported case ‚Üí the existing download-to-view fallback. (`view.officeapps.live.com/op/embed.aspx` is an undocumented public endpoint with a ~10 MB limit + possible rate limits ‚Äî acceptable for best-effort.) `csv` is excluded (download); office extensions live in a preview-local `OFFICE_EMBED_EXT` set since `extensionCategory` has no ppt/pptx.
- **D-P4.4 ‚Äî Object version history (Stage B).** A collapsible footer panel in the preview modal lists S3 object versions (`Cloud/Versions`), **lazy-fetched on expand** (no version call on a normal preview open). Restore (`/Versions/Restore`) + delete (`DELETE /Versions`) are each behind a confirm; the latest version shows a "current" badge and offers no actions (the backend forbids deleting the current version). Both mutations **invalidate the whole owner scope** (`invalidateScope(qc, ownerId)`) on settle ‚Äî restore changes the object (size/ETag) + the version list + the folder listing + usage, and one scope invalidate covers them all (versions are an infrequent action, so the broad refetch is fine). Document versions + diff are deferred to Stage C (they use `Cloud/Documents/Versions(/Diff)` and the modal's `footerSlot` override).
- **D-P4.5 ‚Äî CodeMirror document editor (Stage C1; user choice over a textarea).** Text/code files (`EDITOR_EXT` in `lib/preview`) open in a **CodeMirror 6** editor ‚Äî 11 MIT packages (`codemirror` meta `basicSetup` + `@codemirror/{state,view}` + `lang-{javascript,json,markdown,html,css,python,yaml}` + `theme-one-dark`), **lazy-loaded** via `next/dynamic({ssr:false})` in `DocumentEditorLazy` so CodeMirror lands in its own async chunk (never the initial/preview-host load). Syntax language is resolved by `editorLanguageForName` (pure; the lang `Extension` imports live in the editor chunk). **`.size-limit.json` all-chunks gate raised 480 ‚Üí 820 KB gzip** to admit the lazy editor chunk (measured all-chunks total 788 KB; the editor chunk is ~233 KB gzip ‚Äî `basicSetup` + 7 langs + theme ‚Äî vs the 220 KB pre-impl sketch; the `performance.md ¬ß2` allocation is updated to ~235 KB). Accepted because the chunk is **lazy** (zero initial-load / LCP cost ‚Äî `next/dynamic`); the all-chunks gate is a coarse total-shipped-JS regression catch, not the initial-load metric. Logged per the doc's loosening-needs-a-record rule. **Collaborative-safety lifecycle:** `readContent({includeDraft})` returns content + `ContentHash` (optimistic-concurrency base) + `LockStatus` + `IsDraft` in one call; `acquireLock` on open (`423` ‚Üí read-only + "Locked by {LockedByName}" banner); `extendLock` heartbeat every ~180 s (TTL 300 s) on an interval cleared on unmount; throttled (‚â§1/10 s) `saveDraft` on edit; explicit `updateContent` commit with `ExpectedContentHash` (`409` ‚Üí non-blocking "changed elsewhere ‚Äî Reload" banner, the user's text kept as a draft); `releaseLock` unconditionally on editor unmount; `discardDraft` after a successful commit. Lock/draft calls set `suppressErrorToast` (their `423`/`429` are caller-handled, not errors); commit lets generic errors toast + `409` pass through. **Unsaved-changes guard:** a feature-local `editor.store` (editor registers `{dirty, save, discard}`); the modal's `close()` consults it and shows a Save/Discard/Cancel `AlertDialog` (Cancel aborts ‚Äî the controlled `open` stays). **Tab close = `pagehide`/`visibilitychange` best-effort `saveDraft`, never `beforeunload`** (the secure-folder-lifecycle rule). Commit invalidates the owner scope. **No Idempotency-Key** (the Documents factory exposes none; not in the Move/Delete/Complete set). The editor owns its footer chrome, so the object `VersionHistoryPanel` is suppressed for `viewerKindForName === "editor"`; **document versions + diff are Stage C2** (`Cloud/Documents/Versions(/Diff)` ‚Äî the backend computes the diff, so C2 needs no diff library, just a hunk renderer in the `footerSlot`).
- **D-P4.6 ‚Äî `Cross-Origin-Embedder-Policy` removed (preview iframes were blocked).** Symptom: in dev, the PDF (`cdn.storage.umutk.me`) and Office (`view.officeapps.live.com`) preview `<iframe>`s showed "refused to connect" ‚Äî **not** a CSP issue (CSP is prod-only + `Report-Only`; verified the CDN object returns `206 application/pdf` with **no** `X-Frame-Options`/`Content-Disposition`). Root cause: `STATIC_HEADERS` set `Cross-Origin-Embedder-Policy: credentialless` on **every** response (incl. dev ‚Äî only HSTS/CSP are behind the prod guard), and **COEP blocks cross-origin iframes**: `credentialless` relaxes no-cors *subresources* (CDN images) but **not nested iframes**, which must themselves send COEP ‚Äî the CDN and Microsoft don't. Fix: **drop COEP entirely** (`lib/security/headers.ts`); keep `COOP: same-origin` + `CORP: same-site`. The app uses no `SharedArrayBuffer`/cross-origin isolation, so COEP gained nothing, and it is fundamentally incompatible with the cross-origin preview iframes Phase 4 requires (`security-headers.md` ¬ß5/¬ß6 had an internal contradiction ‚Äî ¬ß5 COEP vs ¬ß6 iframed PDF/office ‚Äî now resolved). This also means the Stage A PDF (native CDN iframe) + Stage B Office embed work without a blob/proxy rewrite. The Playwright header smoke (`tests/security/headers.spec.ts`) dropped its COEP assertion. **Office still needs an HTTPS embedder + a publicly-fetchable doc, so it validates from a deployed HTTPS origin, not `http://localhost`.**
- **D-P4.8 ‚Äî Document version history + diff + restore (Stage C2; closes Phase 4).** A collapsible footer panel for editor files (`DocumentVersionsPanel`, sibling of the Stage-B object `VersionHistoryPanel`), lazy-fetched on expand. **Backend gap fixed first:** `GET Cloud/Documents/Versions` was emitting `void` in the generated client because its controller handler lacked the `@ApiSuccessResponse(...)` decorator + return-type annotation every sibling has (the object `Cloud/Versions` endpoint had it). Rather than hand-roll a DTO (rule #2), the fix mirrors the object endpoint in `nestjs-storage` ‚Äî `@ApiSuccessResponse(CloudVersionListResponseModel)` + `Promise<CloudVersionListResponseModel>` on `ListVersions`, wrapping the service's bare `CloudVersionModel[]` into `{ Versions, Key }`. The document version list **reuses the object `CloudVersionModel`/`CloudVersionListResponseModel`** (same `CloudVersionService.ListVersions` ‚Äî lists only non-current versions, newest first), so the panel is a near-clone. Regenerated the committed client; the **only** diff was `CloudDocumentsApi.listVersions` `void ‚Üí CloudVersionListResponseBaseModel`. **Diff needs no library** ‚Äî the backend computes it: `Cloud/Documents/Versions/Diff` (`{sourceVersionId, targetVersionId}`, `"current"` = live sentinel) ‚Üí `DocumentDiffResponseModel.Hunks` (`{OldStart,OldLines,NewStart,NewLines,Lines}` where `Lines` are raw `+/-/ `-prefixed unified-diff strings) + `Stats`. `DiffView` parses the leading marker and colors lines via **semantic tokens** (`success`/`destructive`/`muted`); diffs are lazy per expanded row (`useDocumentDiff`, `suppressErrorToast` ‚Üí inline error). **Restore** is behind a confirm worded "replaces the current content and reloads the editor ‚Äî unsaved edits lost"; on success it awaits `invalidateScope(qc, ownerId)` then bumps a new `editor.store` **reload signal** (`reloadNonce`/`requestReload`) ‚Äî the open editor watches it and **re-seeds in place via its existing `onReload`** (the 409-reload path: dispatch-replace + `resetBase` + clear-dirty), so the **edit lock is kept** (no remount/lock-churn). Delete just invalidates. **Lock-gating:** the editor publishes `canEdit` to `editor.store`; the panel hides restore/delete when the doc is read-only (locked by another user / lock lost) but keeps the diff viewable (a read-only GET). **Idempotency-Key:** `deleteDocumentVersion` (`DELETE`) gets one automatically via the interceptor's **catch-all-`DELETE`** rule (`service/interceptors/idempotency.ts`); `restoreDocumentVersion` (`PUT /Versions/Restore`) carries none and is correctly outside the protected set ‚Äî both safe (neither auto-retries). **No `.size-limit.json` change** ‚Äî no new deps (the hunk renderer is plain JSX inside the existing budget).
- **D-P4.9 ‚Äî Escape-close vs. the focus-stealing office embed (2026-06-15; addendum to D-P4.3).** A focused cross-origin office embed (`view.officeapps.live.com`) swallows keyboard events, so Radix's Escape-to-close silently failed until focus left the frame ‚Äî the user had to press Escape ~3 times. Fix: `OfficeViewer` uses a `useReleaseIframeFocus(ref)` hook ‚Äî a `window` blur listener that, when `document.activeElement === the iframe`, hands focus back to the document (a deferred `iframe.blur()`), so the next Escape reaches Radix and closes the lightbox. The embed is **read-only**, so releasing focus costs nothing (mouse scroll/click still work); it's purely a keyboard-routing fix. Covered by `tests/preview/office-viewer.test.tsx` (focus-release via a stubbed `document.activeElement`; no-op when not focused). (commit 880a1dd.)

## Phase 5 implementation notes (2026-06-14)
- **D-P5.1 ‚Äî Secure folders is a leaf feature `features/secure-folders/`; one-way `storage ‚Üí secure-folders`** (mirrors preview‚Üístorage). It owns the LOGIC (store, hooks, mutation wrappers, getter, lifecycle) and imports only `@/service`/`@/lib`/`@/stores`/zustand ‚Äî **never `@/features/storage` or `@/features/auth`** (no cycle, ESLint-boundary enforced). Storage CONSUMES it: the dialogs (`UnlockDialog`/`PassphraseDialog`) + the single `SecureFolderDialogs` **controller** live in storage (they use storage's `SectionedDialog` Zen chrome, so moving them would cycle); storage triggers them via the transient `secureFolderUi` store (`open({ kind, path, ‚Ä¶ })`). `auth/signOutAndCleanup` imports `clearAllSecureFolderTokens` (one-way).
- **D-P5.2 ‚Äî In-memory token store + ancestor resolver (rule #5).** `secureFolders.store.ts` (zustand) holds `tokens: Record<"encrypted"|"hidden", Record<rootPath, {token, expiresAt}>>` ‚Äî **never persisted** (no `persist`/localStorage/sessionStorage/cookie; an **ESLint override on that exact file** hard-bans them ‚Äî **verified firing** via a temp violation; a top-of-file DevTools-introspection accepted-risk comment per A8). `isAncestor(ancestor, path)` (added to `lib/utils/paths.ts`, segment-based) drives `resolveToken(entries, path, nowSeconds?)` ‚Äî nearest ancestor; **expiry-checked only when `nowSeconds` is passed** (the interceptor getter passes `Date.now()/1000` so a stale token is never sent; the React query-key fold omits it ‚Äî reading the clock in render is impure and expiry can't move the key, which only reacts to set/clear). Cleared on **sign-out** (`signOutAndCleanup`) + a real **tab close** (`pagehide` when `!event.persisted`; `visibilitychange` no-op; **never `beforeunload`** ‚Äî A1/¬ß11 supersede the older 5.1 text; bfcache restore keeps the in-memory store).
- **D-P5.3 ‚Äî Interceptor reads the target path from the request, not just the URL.** `service/interceptors/secure-folder.ts` (`/Cloud/*`-scoped) extracts `Path`/`Key` from the query (`config.url`) **and the body** ‚Äî the generated client **JSON-stringifies the body before the interceptor runs**, so `config.data` is a string that must be `JSON.parse`d (incl. nested request-model wrappers) ‚Äî then `getSecureFolderToken(path)` ‚Üí injects `X-Folder-Session`/`X-Hidden-Session`. The real getter is registered from `app/providers.tsx` (the P0 `() => null` seam); `service/` stays a leaf (A6 default no-op).
- **D-P5.4 ‚Äî Encrypted (Stage B).** Passphrase rides the generated **`xFolderPassphrase`** param (`X-Folder-Passphrase` header), never body/options/stored. Unlock/encrypt/decrypt `suppressErrorToast` so the dialog owns the 403 ("wrong passphrase") + a **`genericMessage` (`common.errorGeneric`)** so a non-403 (5xx/offline) isn't a silent dead dialog (silent-failure-hunter fix). **A3 order LOCKED:** `useUnlock.onSuccess` = `setToken` ‚Üí `invalidateScope` (reverse = re-prompt loop; spy-tested). The encrypted token is keyed by the returned **`EncryptedFolderPath`**. Locked-row click ‚Üí unlock dialog ‚Üí navigate in; a `403 FORBIDDEN` listing ‚Üí in-place `FolderLocked` state. `directoryLock` drops the token on confirmed success only. The query-key **token fold** (`storageKeys.directories/objects(ownerId, path, secureToken?)` + `useResolvedToken`) makes unlock(set)/lock(clear) refetch; the token-less invalidate key prefix-matches.
- **D-P5.5 ‚Äî Hidden (Stage C).** **The hidden token is keyed by the REVEAL-REQUEST path (the current browse folder), NOT the returned `HiddenFolderPath`** ‚Äî backend-verified: the listing carries `X-Hidden-Session` on the *parent* request and the backend validates it against each hidden *child*; ancestor-resolution only attaches a token to its keyed path + descendants, so keying at the current folder surfaces the hidden children there (the **opposite** of encrypted unlock). Hide/unhide require the passphrase (set-on-hide / verify); reveal reuses `UnlockDialog mode="hidden"`. **`Shift Shift` (`‚áß‚áß`)** = stateful double-tap detection added to `useShortcutDispatcher` (bare Shift, `!repeat`, not editable/overlay, ~400 ms, reset on any intervening key ‚Äî so typing capitals never triggers), registry-gated + storage-scoped via `useStorageCommands`, **with an accessible command-palette alternative** ("Reveal hidden folders", ‚áß‚áß). **A4 conceal atomicity** is adapted for the shared-token model: conceal is a per-folder backend session-delete that drives re-hide **via refetch** (it does NOT mutate the client token, which is reveal-root-scoped) ‚Äî **2xx** ‚Üí `invalidateScope` + success; **NETWORK** ‚Üí **no** invalidate (the session is still live ‚Üí the folder stays revealed) + "try again"; **5xx** ‚Üí invalidate (server truth) + "may not have completed" (silent-failure-hunter confirmed the branch + early-return).
- **D-P5.6 ‚Äî No separate path-marks store (A7 deviation).** The doc's A7 prescribed a persisted `pathMarksCache.store.ts` for badge hints. **Not built** ‚Äî the backend `CloudDirectoryModel.{IsEncrypted,IsLocked,IsHidden,IsConcealed}` flags come fresh on every listing and already drive the badges (`EntryStatusChip`) + the menu gating, so a client marks-cache is unnecessary. A7's intent (tokens never commingled with persisted marks) is **satisfied trivially**: the only secure store is the in-memory token store. **A5 (no `FILE_*` socket events under locked paths)** is a backend contract (lifecycle ¬ß15 MVP: secure folders don't emit live events; changes surface on the next refetch) ‚Äî not a frontend-verifiable item; deferred to backend/live-verify, not gating the frontend.
- **D-P5.7 ‚Äî Session token persists to `sessionStorage` (amends rule #5 + D-P5.2 "never persisted").** *Why:* with the strict in-memory store, a **page refresh** wiped every unlock/reveal ‚Äî the user re-entered the passphrase after every F5, which read as a "hard reload" of their session. A refresh resets all JS memory, so surviving it **requires** some web-storage persistence; there is no in-memory trick. *Decision (user-approved):* persist **only the backend-minted, TTL-bounded session token** to **`sessionStorage`** via zustand `persist` + `createJSONStorage(() => sessionStorage)`, `partialize`d to the token map, with a `merge` that **prunes expired entries on rehydrate** (the clock read is outside React render, so it stays pure). *Invariants kept:* the **passphrase is NEVER stored** (it never enters this state ‚Äî it rides `X-Folder-Passphrase` for one request); **no `localStorage`, no cookie, no cross-tab** (`sessionStorage` is tab-scoped ‚Äî ¬ß12's per-tab guarantee survives unchanged); the ESLint override still hard-bans `localStorage`/`document.cookie` on this file (only `persist`+`sessionStorage` were unbanned). *Lifecycle change:* the `pagehide`/`visibilitychange` handler (`lifecycle/tab-close.ts`) was **removed** ‚Äî `pagehide` fires on refresh too and would have defeated refresh-survival; `sessionStorage`'s own scoping handles tab-close natively, and **sign-out** still clears it (`clearAll()` empties the persisted entry). *Surface trade-off (accepted, ¬ß16):* the token now sits at-rest in `sessionStorage` for the tab lifetime (readable by same-origin script ‚Äî bounded by the strict CSP #12 ‚Äî and DevTools), vs. only while a JS closure held it; the threat stays bounded (TTL-short, tab-scoped, trusted machine, passphrase unstored). Tested in `tests/secure-folders/persistence.test.ts` (rehydrate live / prune expired / write-through / clearAll-wipe / no-localStorage).
- **D-P5.8 ‚Äî Browse auto-refresh + secure-folder TTL re-lock (2026-06-15).** Browse listings now refetch on a **60s interval** (`BROWSE_REFETCH_INTERVAL_MS` in `features/storage/browse/api/browse.config.ts`, consumed by `useDirectories`/`useObjects`) + `refetchOnWindowFocus`, so a folder reflects out-of-band changes (a new upload, a created/moved folder) without a manual reload. **TTL re-lock:** `features/secure-folders/hooks/useSecureFolderExpiry.ts` watches the TTL of the session token covering the *current* folder (via `resolveTokenEntry`) ‚Äî when an **encrypted** token expires while you are inside the folder it `clearToken` + re-opens the unlock dialog (re-prompt for the passphrase; A3 order preserved); when a **hidden** token expires it `clearToken` + toasts + conceals (drives re-hide via refetch, per D-P5.5's shared-token model). A `setTimeout` is scheduled to the token's `expiresAt` (clamped to the 32-bit max), recomputed from primitive `key`/`expiresAt`/`path` deps; the clock is read only at fire time so it stays render-pure. This complements **D-P5.7** (the token is `sessionStorage`-persisted but still TTL-bounded) ‚Äî expiry is enforced at use (the interceptor getter drops a stale token), and now also **proactively at the UI** when it lapses mid-session rather than waiting for the next 403. (touches `secure-folder-lifecycle.md` ¬ß5 + ¬ß15; commit 0761af6.)
