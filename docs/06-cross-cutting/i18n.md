# Cross‑cutting — Internationalization (i18n)

> Two languages **planned**; **English ships at MVP**. Structure now, **no hardcoded copy**. Decision: D7.
> Built in [Phase 0](../01-roadmap/phases/phase-0-foundation.md); enforced everywhere.

## 1. Approach
- A **lightweight dictionary** + `t()` helper in `lib/i18n/` (no heavy framework at MVP; keep the door open for one).
  - Config: `lib/i18n/config.ts`
  - Helper: `lib/i18n/t.ts`
  - Dictionaries: `lib/i18n/dictionaries/<locale>.json` — `en.json` ships at MVP.
- One EN dictionary now; the structure supports adding a second locale without touching components.
- **Every** user‑facing string is a key — even at EN‑only MVP. This is enforced by a **lint rule** against literal JSX
  text / string props that render to the user (see §7).

## 2. Key naming
`area.screen.element[.subelement]` — **all lowercase, dot‑separated**, matching the regex:
```
^[a-z]+(?:\.[a-z]+){1,4}$
```
- Minimum 2 segments (`area.element`), maximum 5 (`area.screen.element.subelement.variant`).
- No camelCase, no underscores, no digits — segments are lowercase ASCII letters only. Long names use additional
  segments, not casing (`emaillabel` is fine because it's clearly a leaf; prefer `auth.login.email.label` when the
  shape is composable).
- Examples:
```
auth.login.email.label
storage.upload.quota.exceeded
storage.conflict.keepboth
common.actions.delete
errors.network.retry
```

### 2.1 Reserved top‑level areas
The first segment is one of the following — adding a new area requires a docs update and a lint allowlist entry:

| Area | Scope |
|---|---|
| `auth.*` | Login, register, password reset, OAuth, verification |
| `account.*` | Profile, settings, security, sessions, subscription |
| `storage.*` | Browser, upload, download, move/copy/rename/delete, share, conflict, quota |
| `preview.*` | Preview shell, media controls, format‑specific viewers |
| `secure.*` | Secure‑folder setup, unlock, lock, recovery, token UX |
| `notifications.*` | Bell, drawer, toast bodies, push, email subjects (where keyed) |
| `teams.*` | Team list, members, invites, roles, workspace switcher |
| `shell.*` | AppShell, sidebar, topbar, command palette, breadcrumbs |
| `legal.*` | Terms, privacy, cookies, consent banners |
| `errors.*` | Typed `ApiError` mapping, generic network/server fallbacks |
| `common.*` | Shared actions (`delete`, `cancel`), shared labels (`name`, `size`), shared statuses |

- `common.*` is the shared UI namespace; `errors.*` is the typed‑error namespace mapped from `ApiError`.
- Keep keys stable; rename via a migration, not silently.

## 3. What gets keyed
- All labels, buttons, headings, placeholders, empty/error/loading copy, toasts, dialog text, validation messages.
- **Not** keyed: data from the API (file names, etc.), developer logs.

## 4. Pluralization & interpolation
- Support `{count}`‑style interpolation in the dictionary helper, e.g. `t('storage.list.items', { count: 3 })`.
- Plurals resolve via [`Intl.PluralRules`](https://developer.mozilla.org/docs/Web/JavaScript/Reference/Global_Objects/Intl/PluralRules)
  against the active locale. Dictionary keys carry a CLDR‑category suffix; `t()` picks the suffix matching the
  resolved category, falling back to `_other`:

  ```json
  {
    "storage.list.items_one":   "1 item",
    "storage.list.items_other": "{count} items"
  }
  ```

  ```ts
  t('storage.list.items', { count: 1 }); // "1 item"
  t('storage.list.items', { count: 7 }); // "7 items"
  ```

  - Required suffix at MVP (en‑US): `_one`, `_other`. Other CLDR categories (`_zero`, `_two`, `_few`, `_many`) are
    accepted by the helper for future locales.
  - The base key (`storage.list.items`) is **not** rendered directly — the helper always selects a suffix; a missing
    `_other` is a lint error.
- Dates/sizes/numbers: format via `Intl`, not hand‑built strings. The wrappers live in `lib/i18n/format.ts`:

  | Helper | Backed by | Use for |
  |---|---|---|
  | `formatNumber(value, opts?)` | `Intl.NumberFormat` | Counts, percentages, currency |
  | `formatBytes(bytes, opts?)`  | `Intl.NumberFormat` + binary units | File sizes, quota |
  | `formatDate(value, opts?)`   | `Intl.DateTimeFormat` | Absolute timestamps, calendar dates |
  | `formatRelativeTime(value)`  | `Intl.RelativeTimeFormat` | "2 minutes ago", "in 3 days" |

  All four read the active locale once and memoize the underlying `Intl` instance. Components import from
  `lib/i18n/format` — they do **not** instantiate `Intl.*` directly.
- MVP locale: **`en-US`** (set in `lib/i18n/config.ts`). A second locale slots in without touching components.
- **RTL ready:** layout uses CSS logical properties (`margin-inline-start`, `padding-inline-end`, `text-align: start`)
  so a future RTL locale is a config flip, not a refactor. See [accessibility §RTL](./accessibility.md).

## 5. Layout implications
- Copy length varies by language — don't hard‑truncate; allow wrap; test with long pseudo‑strings.
- See [typography §6](../03-design-system/foundations/typography.md).

## 6. Acceptance
- A planted hardcoded string fails lint.
- All MVP screens render from keys; switching the (future) locale swaps copy with no component change.
- RTL is **out of MVP** but no layout decision should make a future RTL pass impossible (avoid hard‑coded left/right
  where logical properties suffice).

## 7. Lint enforcement (inline copy heuristic)
- Added in **Phase 0, task 0.12**: a custom `no-restricted-syntax` AST rule in
  [`eslint.config.mjs`](../../eslint.config.mjs) flags any of:
  1. **`JSXText` nodes** whose trimmed content contains **≥ 3 whitespace‑separated words**.
  2. **`JSXAttribute` `Literal` values** (`title=`, `aria-label=`, `placeholder=`, `alt=`, …) whose string contains
     **≥ 3 whitespace‑separated words**.
- The fired message is exactly:
  ```
  Hardcoded string — use t(area.screen.element)
  ```
- **Scope:** the rule runs on every file **except**:
  - `lib/i18n/**` (the helper + dictionaries themselves)
  - `**/*.test.*` and `**/*.stories.*`
  - `tests/**`
- `className` strings and short single‑word literals (icons, separators, units) are exempt by the word‑count shape;
  the rule targets prose, not utilities.
- **Allowlist for proper nouns / brand names** (e.g. "Google Drive", "Yandex Disk"): suppress per‑line with
  ```tsx
  // eslint-disable-next-line no-restricted-syntax -- proper noun: brand name
  <span>Google Drive</span>
  ```
  The trailing `--` reason is **required** (enforced by `eslint-comments/require-description`). Do not whitelist files
  and do not suppress for translatable prose.
- The rule fires as **error** from P0 onward (see folder‑structure plan, rule §7).

## 8. Server vs client `t()` (Next 16.2)
Next 16.2 enforces a hard split between Server Components and Client Components. i18n mirrors the split with **two
entry points** that share **one dictionary**.

### 8.1 The two modules

| Module | Directive | Exports | Used from |
|---|---|---|---|
| `lib/i18n/server.ts` | `import "server-only"` | `t(key, params)` | Server Components, route `metadata`, `sitemap.ts`, `opengraph-image.tsx`, `error.tsx`, route handlers |
| `lib/i18n/t.ts` | `"use client"` | `useT()` React hook, `t(key, params)` | Client Components |

- Both modules read from the same source of truth: `lib/i18n/dictionaries/en.json`.
- **Server** reads the dictionary **synchronously at request time** — no hydration, no network, no client bundle cost.
- **Client** ships the dictionary as JSON in the initial HTML (serialized once by the root layout) and reads from
  memory thereafter; `useT()` is a thin hook over that in‑memory map.

### 8.2 Boundary enforcement
`lib/i18n/server.ts` is **forbidden** in any file marked `"use client"`. Enforced two ways:

- `import "server-only"` at the top of `server.ts` — Next throws at build time if the module is reached from a client
  graph.
- ESLint `no-restricted-imports` rule with a `patterns` entry targeting `lib/i18n/server` from `"use client"` files,
  reinforced by the project's `eslint-plugin-boundaries` setup (see [`data-layer`](../02-architecture/data-layer.md)
  for the boundaries pattern).

Conversely, `lib/i18n/t.ts` is the client entry; server code should not import it (it pulls in React hook plumbing).

### 8.3 Where to use which

| Surface | Entry point |
|---|---|
| `app/**/page.tsx` (Server Component) body | `lib/i18n/server` |
| `app/**/layout.tsx` (Server Component) body | `lib/i18n/server` |
| `app/**/page.tsx` exporting `metadata` / `generateMetadata` | `lib/i18n/server` |
| `app/sitemap.ts` | `lib/i18n/server` |
| `app/opengraph-image.tsx`, `app/twitter-image.tsx` | `lib/i18n/server` |
| `app/**/error.tsx` (Client Component but server‑rendered shell) | `lib/i18n/t` |
| Any `"use client"` component | `lib/i18n/t` |
| Route handlers (`app/api/**/route.ts`) | `lib/i18n/server` |

### 8.4 Signatures
```ts
// lib/i18n/server.ts
import "server-only";
export function t(key: string, params?: Record<string, string | number>): string;

// lib/i18n/t.ts
"use client";
export function useT(): (key: string, params?: Record<string, string | number>) => string;
export function t(key: string, params?: Record<string, string | number>): string;
```
Both honour the §2 key regex, the §4 plural suffix selection, and the §7 lint rule.
